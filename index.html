makefile

MakeFile文件

学习前置：

  1. 明确makefile执行的任务，他是个工具
  2. 可以自己尝试写一写makefile

简介

  1. 解决的事情：编译文件
  2. 依赖树，其中’:’用来表示依赖关系，只有后者已经被存在，前者才能够执行
  3. 命令行： 在依赖树的下边会出现对应的生成命令，会进行执行
  4. 大概有些了解，开始拆解实际的任务

##实战

  1. 命令代替：
    通过例如 AS = gcc 的“赋值”方案直接将gcc 作为一个变量，之后将传入命令行
  2. 命令堆叠：
    通过多次的命令代替，就能够实现命令的替换
  3. 为什么不直接写出命令，而是通过变量的形式？
    这个直接就是c语言中为什么需要变量的原因了，可以轻松替换编译器啊
  4. 为什么会有all ,img 之类的变颜色
    经过尝试，这个表示的就是前后两个文件之间的依赖关系
  5. :冒号的解释：
    1. 前后两者的关系
    2. 确保后者，才能执行命令
  6. 一个makefile中给我的知识：
    1. 首先是定义一堆的变量
    2. 然后是命令
      1. all : img （此时还没有img）
      2. img : asm （又没有asm）
      3. asm : 直接生成asm即可
    3. 仍然是些命令
    4. 很好玩的一个：
      clean 是makefile中定义的，直接通过make clean 运行就好

3.24

date： 21.3.24-25

思考+问题：

  1. cpu，寄存器，内存和对应的汇编指令的关系
    1. cpu无论如何都是处理（process）器，用来进行计算 是这样的吗？还是说有更多的作用，或者说计算机真就是计算机？
    2. 寄存器似乎是cpu的帮手，帮他记账（存储结果），安排生活（存储需要运行的指令）等
    3. 内存的话，就像是他们家的仓库，用来存储 话说也不算，毕竟还有个小弟：硬盘，但是还没看到

笔记：

  1. mov 指令需要 源数据和目标数据两者位数相同，可能这就是为什么c语言不太常用类型转换的原因，或者在某种程度上也是能够进行类型转换的原因，毕竟底层只是几个BYTE的问题（不保证自己的正确性，坑一定会填上的）
  2. ADD 指令，相加，不必多说，但是这里想到前两天看到的一个视频了，非常清晰的解释了加法的实现方式。这是网站
  3. CMP 指令，if的底层实现 与 JE连用，有点if goto的感觉（之前刚看到为什么大家反对goto好像就是因为之前汇编的时候吃的苦头太多了）
  4. INt 指令 不是int 而是打断，好家伙。。。直接宕机
  5. 再谈MOV 之前作者提到的“赋值”现在看来的确是非常正确，尤其是 MOV SS AX 这种句子，把一个寄存器移动到另一个寄存器，的确有点可笑，还是将值进行复制比较好！
  6. 然后是磁盘的一些问题：
    1. 磁区还是能够理解的
    2. 尤其是其中以BYTE为单位进行存储和读取的方式，虽然在一定程度上可能会导致资源的浪费，不过这也是“标准化”的必要性。
    3. 所以，博客的文风和规约也会不断改变，然后确定一种稳定且更加合适（高效且高笑）的风格
  7. makefile 文件：
    1. 之前看过cmake，感觉就有点像是老板对员工讲话，怎么做怎么做
    2. 现在看到makefile之后，就能够感受到这种感觉了，其实本身就是一些命令的集合 但是有些小参数还是不懂的，比如-r之类的，就可以通过’-h’查看help命令，好吧，这就是完整的软件，能够考虑到各种情况
    3. make的执行逻辑（好奇了下）
      1. 读取所有的makefile文件（文件中涉及到的）
      2. 需要完成每个makefile之后才会执行当前的文档（有点像栈的感觉，而每个makefile就相当于一个函数）

今日总结：

  1. 自己的blog算是重新搭起来了，但是欠缺的东西还是很多的
    1. 友链，已经有3位大佬找我了，不能再咕咕咕了（主要是我不会写，从头来）
    2. theme，没法deploy，有点草了，明天再试试！

    4. 还有自己的文风，更像是杂谈，效率不是很高，但是我还是比较喜欢这种风格的，因为本人就没什么逻辑
  2. 算是把操作系统的坑开始填了，但是越看发现自己越菜，好好学吧
    尽快推进看书的速度，然后划上一段时间，自己试着写一写或者改一改linux的源码，尝试理解
  3. 得多尝试，现在我的md还是非常单调的，要尽快学，早日写出五颜六色的md

从头开始

重新开始博客之路

为什么是重新？

因为博主太菜，搞混了hexo的建站方法，上个（2个月前）的博客由于咕咕咕，甚至导致自己最后把hexo的配置文件都给删除了（我吐了！）

从哪里开始？

这个之前也有所提及，不过还是再理下思路：

  1. 操作系统
  2. 嵌入式开发
  3. 导航
  4. 控制算法
  5. 机器学习
  6. 计算机网络
  7. web（前端+后端）
  8. 硬件（电子元件和板子）

为什么要这样做？

  1. 最重要的原因：博主是机器人专业的，所以还是应该向硬件有所偏向的，不然连老本都hold不住就丢人了
  2. 其中，大多数的东西比较底层，而个人也是想去尝试从底开发一些东西，（主要是有些东西现在没有看到可行方案，只能自己去尝试了）
  3. 爱好吧，虽然是机器人，但是个人还是倾向学计算机的，无奈菜的一批，考了机器人，那就来吧
  4. 从底层学，无论是c语言还是cpu都能够看的更加全面一点

怎么做，怎么学？

  1. 学习框架
    1. 参考书（只是参考，更像是字典的作用）
    2. 实际上手（最重要的东西）
    3. 知乎的一些补充（虽然知乎的言论参差不齐，但是有些教程还是不错的）
    4. 谷歌（yyds）
  2. ddl规划
    1. 每月的月末（*.28）是实际验收时间，需要上传对应的代码、软件
    2. 每天需要上传git代码

写在最后
